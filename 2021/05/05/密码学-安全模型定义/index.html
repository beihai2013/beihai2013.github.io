密码学-安全模型定义

步骤

  1. 定义安全特性
  2. 描述game

安全模型一般是通过一个游戏 (game) 来定义, 游戏涉及到挑战者 (challenger) 和敌手 (adversary)

两方的交互. 简单地说, 就是挑战者运行一个相应的方案让敌手进行攻击. 安全模型抽象地定义了敌手可

询问的内容, 并形式化定义了挑战者的安全目标. 一个完整的安全模型可以分为以下四个部分进行理解.

• 初始化 (setup): 挑战者初始化整个系统, 保存系统用于应答 (response) 的秘密参数 (比如: 私钥),

并将公开参数 (比如: 公钥) 返回给敌手.

• 询问 (query): 安全模型通过敌手询问和挑战者应答的方式来抽象现实世界中敌手的攻击. 考虑现

实世界中敌手通过攻击可能得到的结果, 安全模型的定义跳过攻击过程, 直接允许敌手向挑战者发

出相应的攻击结果询问, 挑战者需要正确回答敌手的询问. 例如, 对于公钥加密, 如果假设敌手在

现实世界中能够获得任意密文对应的明文, 则在对应的安全模型定义中允许敌手询问密文对应的

明文, 挑战者必须返回正确的解密结果. 敌手询问的内容以及时间的不同代表着敌手能力大小的不

同. 越强的安全模型应该允许敌手做更多更自由的询问. 有些文章在安全模型定义中把敌手的询

问对象 (即挑战者) 描述为谕言机 (oracle). 这种定义描述上稍有差别, 但本质不变.

• 攻击 (attack): 敌手向挑战者发起攻击挑战. 挑战的对象就是挑战者的安全目标. 敌手成功攻破安

全目标即定义为敌手赢得游戏. 安全目标可以分为两大类: 不可计算性的安全目标和不可区分性

的安全目标. 详细的安全目标的定义和安全需求息息相关. 例如, 数字签名安全模型的安全目标是

签名的不可伪造性, 公钥加密安全模型的安全目标是密文的不可区分性.

• 优势 (advantage): 安全模型的最后一步是对优势 ϵ 的定义. 这里涉及到一个非常重要的概念区

分, 即优势和概率. 在安全模型的定义中, 敌手有一个赢得游戏的理想化概率. 例如, 在数字签名的

EU-CMA 安全模型中, 敌手伪造一个签名的理想化概率是 0, 在公钥加密的 IND 模型中, 敌手猜

对挑战密文所对应明文的理想化概率是 1

2 . 优势是对一个攻击算法或一个求解算法, 相较于理想化

概率有多少更多的可能性, 成功攻破一个方案或成功解决一个问题的度量. 而概率是对一个事件发

生的可能性的度量. 有了优势这个概念, 我们就可以用可忽略和不可忽略来区分一个方案安全与

否

概念

security model

​ 针对加密系统的多种攻击方式的抽象。通常被视为challenger和adversary之间的游戏

challenger

​ 一个虚拟的实体，用于与攻击者adversary交互

adversary

​ 攻击者，通常为PPT攻击者

query

​ 攻击者询问信息

response

​ 挑战者回应信息

win

​ 攻击者攻破方案，获得胜利，说明方案是不安全的

adaptive attack

trivial attack

advantage

​ 攻击者攻破方案的概率

strong security model

​ 除了能引起trivial attack的询问，攻击者询问的信息不受限制

weak security model

​ 攻击者被限定只能询问指定的信息

standard security model

​ 被广泛接受的安全模型

攻击能力

假设攻击者为A(lice)，签名者为Bob

Key-only attack

​ Alice只知道Bob的公钥$pk_B$

Message attack

​ Alice在实施攻击前，能够询问一些明文的签名

known-message attack

​ Alice可以知道一个明文集合$M={m_1,\ldots,m_n}$的签名，但是$M$不是Alice选择的

generic chosen-message attack

​ Alice可以知道一个明文集合$M={m_1,\ldots,m_n}$的签名，$M$是Alice选择的，但$M$与Bob的公钥无关

Directed chosen-message attack

​ Alice可以知道一个明文集合$M={m_1,\ldots,m_n}$的签名，$M$是Alice选择的，$M$可以在Alice知道Bob的公钥后生成

Adaptive chosen-mesage attack

​ Alice可以将Bob作为预言机使用，这意味着$M$不仅可以给予Bob的公钥，也可以给予之前已经询问过的明文

破坏方案

a total break

​ 计算出A的私钥

universal forgery

​ 找到一个有效函数f，等价于Bob的签名算法

selective forgery

​ 针对特定消息，伪造一个签名

existential forgery

​ 至少能对一个消息M，伪造合法签名

Digital Signature[10]

$SysGen(1^\lambda)\rightarrow SP$

$KeyGen(SP)\rightarrow (pk,sk)$

$Sign(SP,sk,M)\rightarrow\sigma_m$

$Verify(SP,pk,M,\sigma_M)\rightarrow 1/0$

EU-CMA

如果一个数字签名是EU-CMA安全的，当且仅当对于任意的PPT攻击者$A$和挑战者$C$，在以下游戏中获胜的概率可以忽略

$Setup$. $C$执行$KeyGen(SP)$得到公私钥$(pk,sk)$，并将公私钥发送给攻击者A，保留$sk$用于应对A的询问

$Query$.A自由查询签名，对于任意消息$m_i$，挑战者$C$将签名$\sigma_i\leftarrow Sign(sk,M)$发送给A

$Forgery$.攻击者输出消息-签名对$(m^*,\sigma_{m^*})$，若等式
$$
Verify(SP,pk,m^*,\sigma_{m^*}):
m\notin{m_1,\ldots,m_n}
$$
成立，则攻击者获胜

SU-EMA

$$
Verify(SP,pk,m^*,\sigma_{m^*}):
(m^*.\sigma_{m^*})\notin{(m_1,\sigma_1),\ldots,(m_n,\sigma_n)}
$$

$Setup$. $C$执行$KeyGen(SP)$得到公私钥$(pk,sk)$，并将公私钥发送给攻击者A，保留$sk$用于应对A的询问

Undeniable Signature[14]

只允许合法用户验签的一种签名。

典型应用场景：防止用户复制软件。用户为signer，软件生厂商为verifier

算法定义

$KeyGen(1^\lambda)\rightarrow (pk,sk)$

$Sign(sk,M)\rightarrow\sigma_m$

$Confirm(m,\sigma_m,pk,sk)$ 是一个多项式时间交互协议，证明$\sigma_m$是一个合法签名

$Disavowal(m,\sigma_m,pk)$ 是一个多项式时间交互协议，证明$\sigma_m$不是一个合法签名

EU-CMA

与原始的数字签名类似，但攻击者A还拥有询问Confirm和Disavowal预言机的能力

Designated Verifier Signature[15]

用于保证Bob不将签名发送给第三方Carol。Bob能仿造Alice的签名。Alice能向Carol证明Bob的签名是伪造的，但是不能否认Alice自己的签名。这在Alice和Bob的签名是计算上不可区分的前提下。

$Sign(sk_S,pk_v,m)\rightarrow\sigma$

$Simul(pk_s,sk_v,m)\rightarrow\sigma$

$Vrfy(pk_s,pk_v,m,\sigma)\rightarrow{accept,reject}$

Non-transferable

$(\tau,q_h,q_s,\xi)-forge$是指在$\tau$时间内，运行$q_h$次哈希询问，$q_s$次签名和模拟函数

Group Signature[24]

$KGen(1^\lambda)\rightarrow(gpk,gmsk,gsk)$，其中$gpk$是群公钥，gmsk是群管理者的私钥，gsk是一个n维向量，每一维代表一个用户的私钥$gsk[i]$

$Sign(gsk[i],m)\rightarrow\sigma$

$Vrfy(gpk,m,\sigma)\rightarrow{1,0}$

$Open(gmsk,m,\sigma)\rightarrow{id,\bot}$

完全匿名性 full-anonymity

可切除性exculpability

无连接性 unlinkability

​ 一个实体，看到签名集合后，不能把两个签名与同一个签名者联系起来
